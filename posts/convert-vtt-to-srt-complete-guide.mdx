# How to Convert VTT to SRT — Fix Errors & Export Right

WebVTT is the web standard for subtitles — but the moment you take your video off the web and into a desktop editor, an LMS, or a media server, VTT becomes a problem. Most professional editing software and legacy platforms don't support it. That's when you need to convert VTT to SRT.

This guide covers the conversion itself, the specific fixes required for major platforms, and how to handle the errors that appear along the way.

## Why Convert VTT to SRT?

VTT files are designed for browser-based playback. Outside of that environment, they cause problems:

- Learning Management Systems like Canvas use SRT for accessibility uploads
- AI-generated captions from tools like Whisper, Otter, and Descript often export VTT — but downstream platforms want SRT
- Media servers like Plex and Jellyfin have better SRT support for local playback

## How to Remove the WEBVTT Header for SRT

Every VTT file starts with a `WEBVTT` header line. SRT files have no such header — if it's present in an SRT file, parsers break.

When converting manually in a text editor, you need to:

1. Delete the `WEBVTT` line and any blank lines before the first cue
2. Add a sequential number before each cue block
3. Convert every `.` in timestamps to `,`
4. Remove any VTT cue identifiers, positioning tags, or styling information

This is tedious and error-prone at scale. The [VTT to SRT Converter](/vtt-to-srt-converter) does all of it automatically.

## Fixing VTT Dot Timestamps to SRT Comma Format

The single most common reason VTT files break in legacy software and hardware players (TVs, Plex clients, older DVD players) is the timestamp decimal separator.

VTT uses a dot: `00:00:04.500`
SRT uses a comma: `00:00:04,500`

It's a one-character difference, but it's enough to make an entire subtitle file unreadable to players that strictly expect SRT formatting. A proper converter handles this; a manual find-and-replace can miss edge cases in files with unusual formatting.

## What Happens to VTT Styling During Conversion?

VTT supports inline styling — bold, italic, colour, and positioning. SRT supports a limited subset: bold (`<b>`), italic (`<i>`), and underline (`<u>`). Everything else is stripped during conversion.

If your VTT file uses `<c.colour>` tags, cue positioning, or alignment metadata, these will not survive the conversion to SRT. The text content is preserved; the styling is not. This is a format limitation, not a converter error — SRT simply doesn't support those features.

For most practical use cases (video editing, LMS uploads, media server playback), this loss of styling is acceptable. The words and timing are what matter.

## After Converting: Check Your File

Before using your converted SRT file, do a quick sanity check:

- Open it in a text editor and confirm the `WEBVTT` header is gone
- Check that timestamps use commas, not dots
- Verify that cue numbers are sequential starting from 1
- Look for any leftover VTT metadata tags (`NOTE`, `REGION`, `STYLE`) — these should be removed
